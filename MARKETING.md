# プロダクト設計資料

## キャッチコピー

> **喋るだけで、申し送りが整う。**
> 頭の中の情報を、10秒でSBAR形式に変換するAIツール。

---

## プロダクト概要

「現場の『言った・言わない』をゼロにする。」  
ミルレコは、手が離せない現場（介護・医療・設営など）のスタッフが「申し送り」を素早く・正確に・安全に残すためのWebアプリです。

マイクに向かって話すか、テキストを入力するだけで、AIが自動的に**SBAR形式（Situation / Background / Assessment / Recommendation）**に整理し、そのまま読み上げられる1文要約も生成します。

アカウント不要、インストール不要。スマートフォンのブラウザで即日使えます。

---

## 解決する課題

### 申し送りにはこんな問題がある

| 現場の声 | 実態 |
|---|---|
| 「口頭で伝えたけど、伝わっていなかった」 | 構造化されていない伝達は抜け漏れが起きやすい |
| 「記録を書く時間がない」 | 業務中に文章を書く余裕はほとんどない |
| 「患者名を記録に残すのが怖い」 | 個人情報の取り扱いに常に緊張がある |
| 「電波が悪くてシステムが使えない」 | 地下・古い建物・山間部では通信が不安定 |
| 「声が出せない環境での申し送りができない」 | 夜勤帯・相部屋・静音エリアではマイク入力が困難 |
| 「AIの解析を待っている時間が長い」 | 忙しい現場では数秒の待機も心理的負荷になる |

ミルレコはこれらすべてに答えます。

---

## 設計思想 ４原則

### 1. 現場第一主義（Field First）

> 「使い始めるまでの手間ゼロ」

ログイン不要。アプリを開いたら即録音できます。「音声で入力」「テキストで入力」を対等なタブ切り替えUIで配置し、どんな環境でも片手操作で完結します。

### 2. プライバシー・バイ・デザイン（Privacy by Design）

> 「個人情報はAIが消す、端末に残さない」

AIが自動で固有名詞を検出し、「A氏」「B氏」に置換してから保存します。**音声ファイルは一切保存しません**。保存されるのは匿名化済みのSBAR結果テキストのみです。GDPR・個人情報保護法に対応した設計思想です。

### 3. 止まらないアプリ（Offline First）

> 「電波がなくても、書いた内容は消えない」

オフライン時はテキスト入力データを端末のIndexedDBに一時保存します。電波が復帰した瞬間、またはアプリを次回起動した時に、バックグラウンドで自動解析・保存が再開されます。「せっかく入力したのに消えた」というストレスをゼロにします。

### 4. 道具としてのUI（Tool, not App）

> 「使い方を覚えなくていい」

ネイティブアプリ感覚のボトムナビゲーションと、解析結果が下からポップアップするボトムシートUIを採用。録音中は声に反応する12本の縦棒ウェーブフォームが表示され、「ちゃんと声を拾えているか」がひと目でわかります。

## 🏥 改善の軌跡とその理由

### 1. 記録から「10秒の思考整理」への特化

**内容:** 音声データの保存や詳細な構造化を捨て、SBAR形式のテキスト生成に全振り。

**理由:** 現場スタッフが欲しいのは「後で聞く録音」ではなく、**「今すぐ次の人に伝えるための言葉」**だからです。保存容量やプライバシーを気にせず、使い捨ての道具として極限まで軽くしました。

### 2. 「音声」と「テキスト」の対等なUI

**内容:** 録音ボタンだけでなく、テキスト入力エリアも主役級の配置に変更。タブ切り替えUIで両者を対等に扱います。1000文字制限と残り文字カウンターも追加。

**理由:** 夜勤帯の静かなナースステーションや、相部屋でのケア中など、**「声が出せない状況」**は必ずあります。どんな環境でも「頭の中を整理したい」という欲求に応えるための配慮です。

### 3. 日本語併記のSBARラベル

**内容:** 単なる S / B / A / R ではなく、**S（状況）/ B（背景）/ A（評価）/ R（提案）**のように日本語で役割を明記。各項目に個別のコピーボタンも配置。

**理由:** SBARは優れた型ですが、慣れていないスタッフには「Aって何だっけ？」という一瞬の迷いが生じます。「10秒で終わらせる」ために、脳のCPU（思考）を使わせないデザインにしました。

### 4. ネイティブアプリ感覚のモバイルUI（全面リデザイン）

**内容:** ボトムナビゲーション（録音/履歴/設定）、解析結果のボトムシート、12本ウェーブフォームビジュアライザー、iPhone safe-area対応、タップ遅延ゼロ化、フォーカス時ズーム防止。

**理由:** 医療現場では手袋をしたまま・小さい画面で操作するケースが多い。ネイティブアプリ相当のタップ体験と視覚的フィードバックが現場定着の鍵です。

### 5. リアルタイムストリーミング表示

**内容:** AIの解析結果を一括取得ではなく、**生成されながら順次ボトムシートへ流し込む**ストリーミング方式に変更。1〜2秒でテキストが流れ始め、完了後に構造化SBARへ自動切り替え。

**理由:** AIの処理自体は変わらなくても、「何かが動いている」という視覚的フィードバックがあるだけで体感待ち時間は大幅に短縮されます。「長い…」という不満をUXで解決しました。

### 6. 「逃げられる」キャンセルボタン

**内容:** 解析中のボトムシートに「中止」ボタンを配置。押すと即座にfetchをAbortしてアイドル状態に戻る。60秒タイムアウトも自動設定。

**理由:** 「間違えて押した」「急に呼ばれた」という現場では、立ち往生するUIは致命的です。どこからでも脱出できる設計が信頼につながります。

### 7. 「消えない」オフライン・ファースト

**内容:** テキスト入力は電波がなくても一旦ローカルに保存し、オンライン復帰時に自動でAI解析を回す。音声データは保存しないため、オフライン時は「テキストで入力」に切り替えを促すメッセージを表示。

**理由:** 医療機関は電波の死角だらけです。「せっかく入力したのに送信エラーで消えた」という体験は、アプリを二度と使わなくなる最大の原因になります。これを仕組みで解決しました。

### 8. プライバシー徹底（匿名化プロンプト強化）

**内容:** AIが自動で「A氏/B氏」に置換し、個人名をデータとして残さない。プロンプトをSBAR各項目の医療的定義に合わせて精緻化し、S/B/A/Rの役割混在を防止。

**理由:** 個人スマホ（BYOD）で使う最大の壁は「セキュリティの不安」です。**「最初から個人情報を残さない仕組み」**にすることで、現場への導入ハードルを劇的に下げました。

### 9. 型安全性とバリデーションの徹底（コード品質）

**内容:** APIのリクエストにZodスキーマを導入し、テキスト入力と音声入力を `discriminatedUnion` で型レベルに分離。共有定数（`SBAR_KEYS` / `SBAR_LABELS`）とコンポーネント（`CopyButton`）を一元化し、`any` 型・非nullアサーション（`!`）をコードベース全体でゼロに。クリップボードAPIのフォールバック、IndexedDBエラーハンドリングなど実務水準の堅牢性を確保。

**理由:** 医療用途のアプリで「AIが返したデータが壊れていた」「想定外のフィールドが欠けていた」というサイレント障害は許容できません。型安全性とランタイム検証を組み合わせることで、表示バグや誤った申し送り内容の生成を防ぎます。

## 主な機能

| 機能 | 説明 |
|---|---|
| 音声入力 | マイクボタン1タップで録音開始。停止で自動解析 |
| テキスト入力 | 音声と対等なタブUI。1000文字制限・残り文字カウント付き |
| SBAR自動生成 | S（状況）/ B（背景）/ A（評価）/ R（提案）に構造化。各項目の医療的定義に沿って精緻に分類 |
| ストリーミング表示 | AIが生成しながら結果を逐次表示。体感待ち時間を大幅短縮 |
| 1文要約 | そのまま読み上げられる申し送り文を自動生成 |
| コピーボタン | SBAR項目ごと・全文・1文それぞれをワンタップでコピー |
| 自動匿名化 | 固有名詞をA氏/B氏に自動置換（音声・テキスト両対応） |
| 解析キャンセル | 解析中に「中止」ボタンで即座に中断可能 |
| オフライン保存 | テキスト入力を一時保存、電波復帰時に自動解析 |
| 次の申し送りCTA | 解析完了後に「次の申し送りを始める」ボタンで即リセット |
| 履歴（最新20件） | 専用タブで過去の申し送りを一覧表示。タップで展開・コピー・削除可能 |
| ボトムナビゲーション | 録音/履歴/設定の3タブ。ネイティブアプリ感覚で操作 |

## ターゲットユーザー

**プライマリ**

- 病院・クリニックの看護師・介護士
- 訪問介護・訪問看護スタッフ
- 介護施設のフロアスタッフ

**セカンダリ**

- チームリーダー・主任（引き継ぎ管理）
- 施設管理者（記録品質の標準化）

**ユーザーが抱える状況**

- 常に時間がない
- 両手がふさがっていることが多い
- ICTに不慣れなスタッフもいる
- 個人スマートフォンで使いたい（BYODニーズ）
- 声を出せない環境での申し送りが必要な場合がある

## 技術的信頼性

| 項目 | 内容 |
|---|---|
| フロントエンド | Next.js 14（App Router） + TypeScript 5 |

| ログイン | なし（端末ローカル完結） |
| オフライン対応 | テキスト入力のみ（音声は通信必須） |
| タイムアウト | 60秒（自動キャンセル） |

## ポジショニング

> **「記録ツール」ではなく「思考整理ツール」**

電子カルテは「後で正式に記録する場所」。ミルレコは「頭の中を整理して、次の人に伝えるための道具」です。用途が違うため競合しません。現場のワークフローに**前工程として自然に組み込まれる**ポジションを狙います。
